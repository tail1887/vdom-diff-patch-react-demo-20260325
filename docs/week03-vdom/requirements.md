# 수요 코딩회 과제 요구사항 (정리본)

> 팀 내부·README·발표 준비용으로, 주최 측 안내를 옮겨 적은 문서이다.  
> 최종 기준은 과제 공지 원문이 우선이다.

---

## 목적

- 하루 동안 AI를 도구로 삼아 팀별 프로젝트에 몰입한다.
- **결과물**과 함께 **과정**도 중요. AI가 작성한 코드는 **원리를 설명할 수 있을 만큼** 소화한다.
- 낯선 기능도 Top-down으로 빠르게 풀며 학습 민첩성을 기르고, 현업에 가까운 기술을 직접 다루어 실무 감각을 익힌다.

## 기능 요구사항

### Virtual DOM & Diff

| 항목 | 내용 |
|------|------|
| DOM → VDOM | 브라우저 DOM 정보를 읽어 **Virtual DOM**으로 변환하는 함수 |
| Diff | 두 Virtual DOM을 비교해 **변경된 부분**을 찾는 **Diff 알고리즘** |

### 검증용 웹 페이지

| 항목 | 내용 |
|------|------|
| 영역 | **실제 영역** + **테스트 영역** |
| 버튼 | **Patch**, **뒤로가기**, **앞으로가기** |
| 로드 시 | 실제 영역 샘플 HTML → VDOM 트리로 변환 → 그 VDOM으로 **테스트 영역 렌더** |
| 테스트 영역 | 사용자가 샘플 HTML(또는 동등한 편집 수단)을 **자유롭게 수정** |
| Patch 클릭 시 | 테스트 영역의 **현재 상태**를 다시 VDOM으로 생성 → 이전 VDOM과 **Diff** → **변경 부분만 실제 영역에 렌더** → 새 VDOM을 **State History**에 저장 |
| 히스토리 | 뒤로가기·앞으로가기 시 **히스토리에 저장된 특정 VDOM**으로 이동하고, **실제 영역·테스트 영역 모두** 그에 맞게 갱신 |

### 기술 스택

- **HTML, CSS, JavaScript (Vanilla)** — 빌드 도구 필수는 아님(팀 합의에 따름).

### 학습 vs 구현

- **구현 우선**. AI 활용 적극 허용.
- 다만 **핵심 로직**은 이해하고 설명 가능해야 함.

## 중점 포인트 (발표·문서에 녹이기)

1. **브라우저에서 DOM 다루기** — `Document`, `Window` 등
2. **DOM 변화 감지** — 변화 감지를 위한 브라우저 API
3. **실제 DOM이 느린 이유** — Reflow / Repaint 관점
4. **Virtual DOM** — 구조, 필요한 이유
5. **Diff 알고리즘** — 동작 방식, **최소 변경을 찾기 위한 5가지 핵심 케이스**, 실제 DOM 반영 방법
6. **React** — 실제 DOM 변경 시 Virtual DOM과 Diff가 어떻게 동작하는지(개념·흐름)
