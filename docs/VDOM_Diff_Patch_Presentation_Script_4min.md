# 발표용 대본 (4분 고정) — DOM ↔ VNode · Diff · Patch · React · 테스트

## 0) 오프닝 (0:00~0:10)

“안녕하세요. 오늘은 브라우저에서 DOM을 Virtual DOM으로 바꿔 스냅샷으로 만들고, 이전과 다음의 차이를 diff로 계산한 뒤 patches 목록을 통해 patch로 필요한 부분만 실제 화면에 반영하는 흐름을 데모로 보여드리겠습니다.”

---

## 1) 1단계: DOM → VNode (0:10~1:10)

[화면: 1단계 DOM → VNode]

“지금 왼쪽은 실제 DOM이고, 오른쪽에는 이 DOM을 읽어서 만든 VNode JSON이 보입니다.  
이 단계는 DOM을 그대로 비교하지 않고, 비교하기 쉬운 형태인 VNode로 바꿔 스냅샷을 만드는 과정입니다.  
즉 ‘화면 상태’를 데이터로 옮기는 거예요.  
여기서 중요한 건, 화면에 보이는 구조와 텍스트가 VNode에서는 어떤 형태로 표현되는지입니다.  
그리고 이 스냅샷이 이후 diff 계산의 입력이 되기 때문에, 형식이 일관되면 같은 변화에 대해 같은 종류의 패치로 이어집니다.”

---

## 2) 2단계: VNode → DOM (1:10~1:30)

[화면: 2단계 VNode → DOM]

“여기서는 방금 만든 VNode를 다시 실제 DOM으로 렌더합니다.  
즉 VNode가 ‘화면을 정확히 표현할 수 있는지’ 복원 결과를 확인하는 단계예요.  
렌더된 결과는 샘플과 충돌하지 않도록 id가 제거된 버전을 사용하고, 그래서 화면은 거의 비슷하게 보입니다.  
여기서 복원 결과가 맞으면, 다음 단계의 diff가 의미 있는 비교를 할 수 있습니다.”

---

## 3) 3단계: Diff와 시나리오 (1:30~2:30)

[화면: 3단계 Diff + 오른쪽 패치 목록]

“이제 Diff 단계입니다. 지금부터는 DOM을 직접 바꾸지 않고, 이전 VNode와 다음 VNode의 차이를 패치 목록으로 계산합니다.  
오른쪽의 패치 목록 패널에 그 결과가 보입니다.”

“시나리오는 기본값이 id 제거 복제입니다. 그래서 화면은 거의 비슷하고, 바뀌는 건 id 같은 속성 쪽입니다.  
이 시나리오를 다른 값으로 바꾸면 패치 목록에서도 텍스트나 구조 쪽 변화가 달라지는 걸 확인할 수 있어요.  
셀렉트에서 ‘동일 트리(깊은 복제)’, ‘id 제거 복제’, ‘첫 제목 변경’을 바꿔보면, 화면 변화의 양이랑 patches 변화의 방향이 같이 연결됩니다.”

---

## 4) 4단계: Patch로 실제 반영 (2:30~3:05)

[화면: 4단계 Patch]

“이제 Patch 단계입니다. 초기화를 한 번 누른 뒤, 현재 시나리오 패치 적용 버튼을 누르면 실제 화면에 반영됩니다.”

“여기서 포인트는 id가 실제로 사라지는지입니다. Shadow 안의 DOM이라서 개발자 도구에서 shadow-root를 열어 확인하면 더 확실합니다.  
중요한 건, 지금은 다시 diff를 돌리는 게 아니라 3단계에서 계산한 결과를 그대로 실행해서 ‘계산된 만큼만’ 화면이 바뀐다는 걸 보여주는 단계라는 점입니다.”

---

## 5) 5단계: 워크숍 흐름 (3:05~3:35)

[화면: 5단계 통합 워크숍]

“5단계는 실제 영역과 테스트 영역을 분리한 ui입니다.  
오른쪽 textarea는 다음 상태를 입력하는 곳이고, 실제 영역은 Patch 버튼을 눌렀을 때만 갱신됩니다.”

“Patch를 누르면 textarea 내용을 새로운 VNode로 만들고, 그때 old와 비교해서 diff를 계산한 다음 patch로 실제 영역만 업데이트합니다.  
즉 textarea는 입력이지만, 실제 DOM 갱신은 버튼을 눌러야 일어납니다.  
뒤로/앞으로는 다시 diff를 계산하지 않고 history에 저장된 스냅샷을 되돌려서, 실제 영역과 입력 상태를 같이 맞춰 복원합니다.”

---

## 6) 6단계: 단위테스트로 검증 (3:35~3:55)

[화면: 6단계 단위테스트 PASS/FAIL]

“마지막은 단위테스트입니다. 로드 시 자동으로 실행되고, 필요하면 테스트 실행 버튼으로 다시 돌릴 수 있어요.”

“PASS는 핵심 함수들이 잘 동작했다는 뜻이고, FAIL이 보이면 로그에서 어떤 케이스에서 결과가 달랐는지 바로 찾을 수 있습니다.  
예를 들면 텍스트 변경, 속성 변경, 추가/삭제, 구조 교체 같은 핵심 상황별로 결과를 확인할 수 있어요.”

---

## 마무리 한 문장 (3:55~4:00)

“정리하면 VNode는 스냅샷이고, diff는 차이를 패치 목록으로 계산하며, patch는 그 목록대로 실제 DOM을 최소 변경으로 반영합니다. React도 큰 원리는 같습니다.”
