## 1단계: 첫 마운트

### 먼저 화면에서 할 것

우선 1단계 박스를 보겠습니다.  
렌더 -> VNode -> DOM -> 화면 부착 파이프라인이 보입니다.

여기서 핵심은 이겁니다. 함수가 VNode를 만들고, 그 VNode를 DOM으로 바꾼 뒤 컨테이너에 붙입니다.

즉 이 단계 코드의 역할은,  
첫 렌더를 실제 DOM으로 변환/부착하는 엔진 본체를 보여주는 것입니다.

왜 이게 필요하냐면,  
VNode는 JavaScript 객체라서 그대로는 화면에 안 보입니다.  
그래서 `vnodeToDom`과 `append`가 반드시 필요합니다.

### 코드 보기 열었을 때 짚을 포인트

첫 번째 코드에서는
- `mountIf(document.getElementById(...), function Week4Phase1(){ ... })`
  - 어디에 붙일지 + 어떤 renderFn을 실행할지 연결

두 번째 코드에서는
- `FunctionComponent.prototype.mount`
  - `renderFn()`으로 VNode 생성
  - `vnodeToDom(vnode)`으로 DOM 변환
  - `appendChild`로 실제 화면 반영

이런 식으로 코드 안에 방금 말한 과정이 그대로 들어 있는 것을 확인할 수 있습니다.

### 다음 문제 연결

그런데 앱은 첫 렌더로 끝나지 않습니다.  
버튼 클릭, 입력 같은 사용자 동작이 들어오면 값이 계속 바뀝니다.

문제는 여기서 생깁니다.  
값이 바뀔 때마다 함수를 다시 실행해야 최신 화면을 만들 수 있는데,  
그냥 지역변수를 쓰면 매 실행 때 초기화됩니다.

그래서 2단계가 필요합니다.

## 2단계: 상태와 리렌더 (`useState`)

### 먼저 화면에서 할 것

`+1`, `reset` 버튼을 눌러서 값 변화를 보여줍니다.

### 그대로 읽을 멘트

2단계의 해결책은 훅(Hook)의 일종인 `useState`입니다.

훅은 함수 컴포넌트가 다시 실행돼도 값을 기억하게 해 주는 장치입니다. 

왜 기억이 필요하냐면,  
함수는 다시 실행될 때 지역변수가 새로 만들어지기 때문입니다.  
지역변수만 쓰면 값이 유지되지 않습니다.

그래서 상태는 함수 본문 안의 일반 변수에 두는 게 아니라,  
컴포넌트 인스턴스(함수로 만들어진 개별 객체를 의미한다)의 `hooks` 슬롯에 저장합니다.

첫 번째 코드에서
`var st = useState(0); var n = st[0]; var setN = st[1];`  
이렇게 읽기 값과 쓰기 함수를 분리해 둔 이유도,  
상태 읽기와 상태 변경의 흐름을 명확히 하기 위해서입니다.  
읽기는 `n`으로 현재 스냅샷을 확인하고, 변경은 반드시 `setN(...)`을 통해서만 일어나게 하면  
어디서 상태를 바꿨는지 추적이 쉬워지고 렌더링 재실행 시점도 일관되게 관리할 수 있습니다.
아래 이벤트 함수를 보면 클릭할 때마다 값이 1씩 증가합니다.  
실제 데모에서도 기본값이 리렌더 후 사라지지 않고 유지되는 것을 확인할 수 있습니다.

두 번째 코드에서
- `useState` 내부를 보면
  - `hookIndex`는 “이번 렌더에서 몇 번째 훅 호출인지”를 가리키는 번호입니다.  
    그래서 `useState`가 호출될 때마다 그 번호의 `hooks` 슬롯을 읽습니다.
  - 첫 렌더에서는 해당 슬롯이 비어 있으므로 초기값으로 슬롯을 생성합니다.
  - 다음 렌더부터는 같은 순서의 호출이 같은 슬롯을 다시 읽어서, 이전 상태가 유지됩니다.
  - `setState`는 슬롯 값을 바꾼 뒤 즉시 DOM을 건드리지 않고 `scheduleUpdate`로 “업데이트 예약”만 합니다.
- 업데이트 루프(실행 순서와 의미):
  - `setState`  
    -> 상태 저장소(`hooks` 슬롯) 값 변경
  - `scheduleUpdate`  
    -> 업데이트를 큐에 넣어 한 번에 처리할 준비
  - `update`  
    -> 컴포넌트를 다시 실행해 새 VNode 트리 생성
  - `diffVNode`  
    -> 이전 트리와 새 트리 비교해 patch 목록 계산
  - `applyPatches`  
    -> 계산된 patch만 실제 DOM에 반영
즉, 업데이트 흐름이 실제 코드에서도 같은 순서로 동작하는 것을 확인할 수 있습니다.


### 다음 문제 연결

여기까지 오면 “값이 바뀌면 다시 그린다”는 문제는 해결됩니다.  
그런데 또 문제가 남습니다.

리렌더는 필요한데, 파생 계산까지 매번 다시 하면 비효율적입니다.

그래서 3단계에서 “필요할 때만 재계산”을 도입합니다.

---

## 3단계: 파생값 최적화 (`useMemo`)

### 먼저 화면에서 할 것

- `tick +1`만 눌러서 리렌더는 나지만 계산값이 유지되는지 확인
- `count +1`을 눌러서 계산값이 바뀌는지 확인

여기 3단계의 핵심은 `useMemo`입니다.

`useMemo`는 deps가 바뀔 때만 계산을 다시 하고,  
안 바뀌면 이전 계산 결과를 재사용합니다.

예를 들어 첫 번째 코드를 보면
`var derived = useMemo(function(){ return count * 10 + 1; }, [count]);`  
여기서 `[count]`는 비교 대상 목록입니다.


두 번째 코드를 보면

- `shallowDepsEqual`는 이전 deps와 현재 deps 비교
- deps 동일 -> 캐시 반환
- deps 변경 -> `factory` 재실행 후 캐시 갱신

즉

`count`가 바뀌면 재계산하고,  
`tick`만 바뀌면 캐시를 재사용합니다.

즉, `useMemo`는 이벤트 처리 도구가 아니라 계산 최적화 도구입니다.

### 다음 문제 연결

여기까지는 “렌더 중 계산” 문제를 해결한 겁니다.  
그런데 앱에는 렌더 밖에서 해야 할 일도 있습니다.

예를 들면 로그 남기기, 외부 DOM 수정, 구독/해제, 타이머 정리 같은 작업입니다.

그래서 4단계가 필요합니다.

---

## 4단계: 렌더 뒤 작업 (`useEffect`)

### 먼저 화면에서 할 것

- `theme 토글`, `msg` 버튼을 눌러 로그 변화를 보여줍니다.
- 아래 타임라인이 최신순으로 쌓이는 것을 보여줍니다.

4단계의 핵심은 `useEffect`입니다.

`useEffect`는 렌더 함수 안에서 “등록”만 하고,  
실행은 DOM 반영이 끝난 뒤 마이크로태스크(microtask)에서 일어납니다.

그래서 이 단계의 메시지는 이겁니다.  
“화면을 그리는 일”과 “그린 뒤 해야 할 일”을 분리합니다.

지금 데모 로그를 보면 보통 순서가 이렇게 나옵니다.  
`render -> cleanup(이전) -> effect(현재)`

즉, deps가 바뀌어 effect를 다시 실행할 때는  
이전 effect의 cleanup을 먼저 호출하고,  
그 다음 현재 effect를 실행합니다.

### 코드 보기 열었을 때 짚을 포인트

- effect 본문: 외부 DOM 텍스트 갱신(부수 효과)
- 반환 함수: cleanup
- `flushEffects`:
  - `queueMicrotask`로 예약되어 실행
  - pendingRun인 effect만 처리
  - 이전 cleanup 먼저 실행 후 effect 실행

### 다음 문제 연결

이제 상태/계산/부수효과는 갖췄습니다.  
하지만 컴포넌트를 자식으로 나누기 시작하면 구조 문제가 생깁니다.

상태를 여기저기 두면 누가 진짜 값을 갖는지 헷갈립니다.  
그래서 5단계에서 상태 소유권을 정리합니다.

---

## 5단계: 자식은 받기만, 상태는 루트만 (Lifting State Up)

### 먼저 화면에서 할 것

5단계 데모의 시각 블록을 보겠습니다.

- 왼쪽: 루트 state 원본 (`count`, `label`)
- 가운데: `props 전달 ->`
- 오른쪽: 자식은 받은 props만 사용
- 아래 `ChildPreview`가 자식이 실제로 받은 props를 그대로 보여줌

버튼을 누르면 왼쪽 state가 바뀌고, 오른쪽 표시가 따라 바뀝니다.

### 그대로 읽을 멘트

5단계 핵심은 두 줄입니다.

1. 상태는 루트에서 소유한다.  
2. 자식은 props를 받아 표시만 한다.

왜 이렇게 하냐면,  
여러 자식이 같은 데이터를 볼 때 상태가 분산되면 동기화가 쉽게 깨지기 때문입니다.

루트 한 곳에서 상태를 관리하면,  
변경 경로가 단순해지고 디버깅이 쉬워집니다.

코드에서 `Header(props)`와 `ChildPreview(props)`는  
둘 다 props만 받아 VNode를 반환하는 표시 전용 자식입니다.

그리고 `h(type, props)` 내부에서  
`typeof type === 'function'` 분기일 때 `activeComponent`를 잠깐 끄는 이유는,  
이 과제의 제약인 “자식 훅 금지(무상태 자식)”를 런타임에서 강제하기 위해서입니다.

### 코드 보기 열었을 때 짚을 포인트

- 루트에서만 `useState/useMemo/useEffect` 사용
- 자식 호출은 `h(Header, props)`, `h(ChildPreview, props)`
- 자식은 받은 props 그대로 표시
- 상태 변경 버튼은 루트에만 존재

### 5단계 한 줄 결론

자식은 props-only, 상태는 루트 단일 소유.

## 부록 A — 현장 Q&A 짧은 답변

- Q. `container`는 서버 데이터인가요?
  - A. 아니요. `document.getElementById(...)`로 찾은 페이지 DOM입니다.
- Q. `return h(...)`는 DOM 반환인가요?
  - A. 아니요. VNode 객체 반환입니다. DOM 변환은 `vnodeToDom`이 담당합니다.
- Q. `Week4Phase1`은 무엇인가요?
  - A. renderFn입니다. 호출되면 VNode 트리를 계산해 반환합니다.
- Q. `useMemo`와 `useEffect`는 둘 다 deps를 보는데 뭐가 다른가요?
  - A. `useMemo`는 렌더 중 계산/캐시, `useEffect`는 렌더 후 부수 효과 실행입니다.
- Q. deps는 클릭이 직접 바꾸나요?
  - A. 클릭은 보통 `setState`를 호출하고, deps 변경 판단은 다음 렌더에서 이전/현재 값 비교로 결정됩니다.
- Q. cleanup은 왜 필요한가요?
  - A. 이전 effect가 만든 바깥 상태(구독, 타이머, 핸들러 등)를 다음 effect 전에 정리하기 위해 필요합니다.

## 부록 B — 발표 동선 체크리스트

- 1단계: 코드 보기 열고 `container`, `renderFn`, `VNode` 용어 정확히 짚기
- 2단계: `+1`, `reset` 직접 눌러 상태 유지 확인
- 3단계: `tick +1` 먼저, `count +1` 다음 순서로 보여주기

- 5단계: 시각 블록(루트 state -> props 전달 -> 자식 표시) 먼저 보여주고 코드로 내려가기
- 마지막: 테스트 버튼으로 자동 검증까지 마무리
